# OR 모델 Ralph Loop 최적화 리포트

생성 시각: 2026-05-21T02:15:30Z

## 반복 목표

기존 평가에서 드러난 한계를 그대로 인정하는 데서 멈추지 않고, 각 한계의 본질에 맞는 최소 알고리즘 수정을 적용한 뒤 같은 평가 하네스로 전후 변화를 확인했다.

## 적용한 최적화

<details>
<summary>반복 1. 요청 시점 NEMC 병상 재호출</summary>

### 의도
저장 스냅샷과 현재 병상 상태가 크게 달라지는 문제는 모델 문제가 아니라 데이터 시간성 문제다. 따라서 저장 스냅샷을 더 잘 해석하려 하지 않고, 추천 직전에 NEMC 실시간 병상을 다시 가져오도록 바꿨다.

### 변경
- `/api/or/recommendations`는 기본적으로 요청 시점 NEMC 병상 재호출을 수행한다.
- `refresh_capacity=false`를 명시한 경우에만 저장 스냅샷을 사용한다.
- `/api/or/hospitals?refresh=true`도 재호출 병상 데이터를 반환한다.

### 검증
API smoke test에서 `candidatePolicy=request_time_nemc_live_capacity`, `capacityRefresh.enabled=true`, `activeRows=51`을 확인했다.
</details>

<details>
<summary>반복 2. 서울형 이송시간 추정기</summary>

### 의도
기존 이송시간은 직선거리와 고정 속도만 사용해 실제 도로 이동을 과소추정할 위험이 컸다. 외부 경로 API 없이도 최소 개선을 하기 위해, 위도/경도 기반 우회거리와 서울 도심/한강 횡단/거리별 속도 차이를 반영했다.

### 변경
- 직선거리 기반 고정속도 식을 `seoul_urban_detour_v2` 추정기로 교체했다.
- 우회거리 계수, 도심 접근 지연, 한강 횡단 패널티, 거리 구간별 구급차 속도를 반영했다.

### 결과
이송시간이 더 보수적으로 산정되면서 20분 제한 케이스 일부에서 엄격 제약 만족 후보 없음이 발생했다. 이는 나쁜 회귀라기보다, 기존 시간이 과소추정되어 감춰졌던 feasibility gap이 드러난 것이다.
</details>

<details>
<summary>반복 3. 병상/자원 패널티 재보정</summary>

### 의도
병상 1개 리스크는 반드시 반영해야 하지만, 병상 버퍼가 거리보다 과도하게 순위를 지배하면 현장적으로 납득하기 어렵다. 따라서 병상/자원 패널티를 낮추고, 여유시간 증폭도 완화했다.

### 전후 비교
| 항목 | 최적화 전 | 최적화 후 |
|---|---:|---:|
| 패널티 격자 최대 1순위 변경 수 | 25/40 | 18/40 |
| 고불안정 계수 조합 수 | 5 | 0 |
| 병상 버퍼 제거 시 1순위 변경 수 | 20/40 | 15/40 |
| 자원 여유 제거 시 1순위 변경 수 | 12/40 | 2/40 |
| 기준모델 대비 1순위 변경 수 | 26/40 | 14/37 |
| 바뀐 경우 V2가 더 낮은 병상 수를 고른 수 | 3 | 0 |

### 해석
계수 민감도는 크게 줄었다. 특히 V2가 기준모델보다 더 낮은 병상 수를 고르는 사례가 0이 되었으므로, 병상 버퍼 목적은 더 일관되게 유지된다.
</details>

<details>
<summary>반복 4. 엄격 제약 실패 fallback 진단</summary>

### 의도
현실적인 이송시간을 쓰면 모든 케이스에서 엄격 제약 만족 후보가 존재한다고 가정할 수 없다. 이때 모델이 조용히 실패하면 운영적으로 위험하다.

### 변경
추천 결과에 다음 진단을 추가했다.
- `strictFeasibleCount`
- `relaxedFallbackUsed`
- `constraintDiagnostics.violationCounts`
- `constraintDiagnostics.minimumTransportTimeForNonTimeFeasibleCandidateMin`
- `constraintDiagnostics.dominantRelaxationHint`

### 해석
hard constraint는 유지하되, strict candidate가 없을 때 어떤 제약이 병목인지 구조적으로 설명할 수 있게 됐다. 이로써 “추천 실패”가 아니라 “제약 충돌 진단 + 완화 후보 제시”가 가능해졌다.
</details>

## 남은 Gap

<details>
<summary>남은 Gap 1. 실제 도로 경로 API 또는 실제 이송시간 데이터</summary>

서울형 이송시간 추정기는 직선거리 고정식보다 낫지만, 실제 도로 네트워크 기반 시간은 아니다. 연구 claim을 강하게 하려면 카카오/네이버/OSRM 등 경로 API 또는 실제 구급 이송시간 데이터를 붙여야 한다.
</details>

<details>
<summary>남은 Gap 2. 20분 제한 케이스의 feasibility</summary>

현실적인 이송시간을 적용하자 3개 시나리오에서 엄격 제약 만족 후보가 없었다. 이는 모델 실패라기보다, LLM이 추출한 `max_transport_time_min`과 병원 자원 hard constraint가 동시에 너무 강할 수 있음을 의미한다. 현재는 fallback 진단을 추가했으므로 운영적으로는 닫혔지만, 연구적으로는 triage protocol을 더 명확히 정의해야 한다.
</details>

<details>
<summary>남은 Gap 3. HIRA 정적 병원 프로필</summary>

의사 수, 전문의 수, 총 병상 수, 중환자실 병상 수가 아직 모두 비어 있다. 따라서 병원 규모/인력 안정성 보정은 아직 연구 주장에 포함하면 안 된다.
</details>

## 현재 가능한 연구 주장

현재는 다음 수준의 주장이 가능하다.

> 서울 응급의료기관의 요청 시점 NEMC 실시간 병상과 병원 자원 정보를 사용하고, 위도/경도 기반 서울형 이송시간 추정과 capacity-buffer objective를 결합해 단순 최단거리 기준보다 병상 소진 리스크를 더 일관되게 반영하는 응급 이송 병원 추천 정식화를 제안했다. 40개 시나리오에서 계수 민감도, 이송시간 충격, LLM 파라미터 흔들림, 병상 재호출 드리프트를 검증했고, 엄격 제약 충돌 시 fallback 진단을 제공한다.

아직 하면 안 되는 주장은 다음이다.

- 실제 도로 경로시간을 완전히 반영했다.
- 의사 수/총 병상 수 기반 병원 규모 안정성을 반영했다.
- 실제 환자 outcome 기준으로 최적임이 검증됐다.
